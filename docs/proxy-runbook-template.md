## 6. Инциденты: playbooks

### 6.1 Симптом: сервис недоступен (P1)

#### Диагностика

1. Проверить внешний healthcheck.
2. Проверить статус процесса/контейнера.
3. Проверить порт и firewall.
4. Проверить логи последних 200 строк.
5. Проверить ресурсы хоста (CPU/RAM/disk).

#### Действия

1. Перезапустить сервис.
2. Если не помогло - rollback на предыдущую рабочую версию.
3. Если проблема на уровне хоста - reboot по согласованию.
4. Если IP заблокирован - переключение на резервный endpoint (если есть).

#### Критерий восстановления

- Внешний healthcheck green минимум 10 минут.
- Error-rate вернулся к норме.
- Нет новых рестартов.

#### Пост-инцидент

- Заполнить RCA-шаблон (см. раздел 11).

### 6.2 Симптом: высокий error-rate

#### Диагностика

1. Проверить, где ошибки: auth / upstream / network / timeout.
2. Сверить момент роста с деплоем/изменениями.
3. Проверить latency, соединения, ресурсы.
4. Проверить доступность внешних зависимостей.

#### Действия

1. Откатить последний релиз (если корреляция с релизом).
2. Временно увеличить лимиты ресурсов (если упираемся в хост).
3. Ограничить всплеск подозрительного трафика (rate-limit / firewall rules).
4. Эскалация владельцу сервиса.

### 6.3 Симптом: частые рестарты процесса

#### Диагностика

1. Посмотреть причину exit code.
2. Проверить OOM / out-of-disk.
3. Проверить валидность конфигов/секретов.
4. Проверить изменения в окружении.

#### Действия

1. Вернуть последний рабочий конфиг.
2. Откатить версию.
3. Устранить ресурсный дефицит.
4. Подтвердить стабильность 15+ минут.

## 7. Деплой / Rollback

### 7.1 Деплой (стандартная процедура)

1. Проверить чистоту состояния и доступность мониторинга.
2. Применить релиз `<команда>`.
3. Проверить healthcheck локально и извне.
4. Проверить метрики 10-15 минут.
5. Зафиксировать версию в changelog.

### 7.2 Rollback

1. Выбрать предыдущую стабильную версию `<version/tag>`.
2. Выполнить откат `<команда>`.
3. Повторить post-deploy проверки.
4. Отметить причину отката.
