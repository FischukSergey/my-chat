# Known Limitations — Sprint 1

Ограничения, принятые осознанно в рамках MVP Sprint 1.
Каждое из них является кандидатом на устранение в следующих спринтах.

---

## Auth

**Нет реальной аутентификации.**
`POST /api/v1/auth/login` принимает произвольный `user_id` из тела запроса без какой-либо проверки.
Любой клиент может «залогиниться» под любым идентификатором. Подходит только для dev/demo окружения.

**JWT не инвалидируется при logout.**
`POST /api/v1/auth/logout` отвечает 204 без дополнительных действий — у сервиса нет хранилища активных токенов (blacklist). Существующий access-токен продолжает работать до истечения срока действия.

**Нет автоматического refresh на клиенте.**
Debug UI не обновляет токен автоматически при истечении. Нужно нажать «Login» повторно.

---

## WebSocket

**Нет heartbeat (ping/pong).**
Сервер не отправляет проверочные фреймы, чтобы обнаружить «мёртвые» соединения. При обрыве сети TCP-сокет может висеть неопределённо долго.

**Нет доставки пропущенных событий при reconnect.**
Если пользователь был офлайн, события (`message_new`, `message_delivered`, `message_read`) не буферируются. После повторного подключения клиент не получит события, отправленные в его отсутствие. Для актуального состояния нужно делать REST-запрос к `GET /api/v1/dialogs/{id}/messages`.

**`message_delivered` отправляется только при онлайн-получателе.**
Событие `message_delivered` генерируется в момент отправки сообщения, только если получатель подключён по WebSocket. Нет механизма отложенной доставки.

**Нет ограничений на количество соединений одного пользователя.**
Один `user_id` может открыть произвольное число WebSocket-соединений — все они получат события одновременно.

---

## API

**Нет rate limiting.**
Эндпоинты не ограничивают частоту запросов. Злоупотребление возможно.

**Нет пагинации через offset, только cursor (`before`).**
`GET /api/v1/dialogs/{id}/messages` поддерживает только курсорную пагинацию по `before` (timestamp). Offset-пагинация не предусмотрена.

**Диалог создаётся клиентом с явным `dialog_id`.**
Клиент передаёт `dialog_id` при первом `POST /api/v1/dialogs/{id}/messages`. Сервер использует `GetOrCreate` — конфликт невозможен, но клиент обязан заранее знать или сгенерировать UUID диалога.

---

## Инфраструктура

**Только Docker Compose для локальной разработки.**
Нет Helm-чартов, манифестов Kubernetes и production-конфигурации.

**Миграции запускаются при каждом старте сервиса.**
`Migrate()` выполняется при каждом запуске `main-service`. Используется `CREATE TABLE IF NOT EXISTS`, что безопасно, но не является production-подходом (нет версионирования миграций через goose/migrate).

**Нет observability.**
Нет трассировки (OpenTelemetry), метрик (Prometheus), алертов. Только структурированные логи через `slog`.
